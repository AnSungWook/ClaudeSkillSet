# 토큰 최적화 가이드

하네스 기반 워크플로에서 토큰을 효율적으로 사용하는 방법.

> **핵심 원칙: 토큰 절약 ≠ 품질 절약.**
> 검수 1회를 줄이는 것(10K~30K 토큰)이 thinking 토큰을 제한하는 것보다 훨씬 효과적이다.

---

## 왜 하네스가 토큰을 절약하는가

### Plan 모드의 ROI

```
Plan 없이 구현 → 자유 생성 → 검수 6~8회 × (읽기 + 수정 + 재검수)
Plan 후 구현  → 구속 생성 → 검수 2~3회 × (읽기 + 미세 수정)
```

- 검수 1회 = 최소 10K~30K 토큰 (코드 읽기 + 분석 + 수정 + 확인)
- 검수 6회 차이 = **60K~180K 토큰 절약**
- Plan 모드 자체 비용(~5K~10K) 대비 압도적 ROI

### 스킬의 프로그레시브 로딩

스킬은 frontmatter(`name`, `description`)만 매 턴 노출된다.
body 내용은 **호출 시에만** 컨텍스트에 로드.

→ 스킬이 20개여도 사용하지 않는 스킬의 body는 토큰을 소비하지 않는다.
→ 단, 스킬 목록 메타데이터는 항상 로드되므로 극단적으로 많은 스킬은 오버헤드가 있다.

### ADR이 재작업을 줄이는 구조

ADR = "왜" 기록 → AI가 맥락을 이해 → 잘못된 판단 감소 → 재작업 감소 → 토큰 감소.
"이렇게 해라"만 있으면 → AI가 상황에 따라 무시 → 검수에서 걸림 → 토큰 증가.

## Tier 1: 설정으로 즉시 절약

### 1. `.claudeignore` 설정

Claude가 코드베이스 탐색 시 불필요한 파일을 읽지 않도록 제외. **설정 한 번으로 얻는 무료 절약.**

```
# 빌드 산출물
**/build/
**/dist/
**/out/

# 의존성
node_modules/
.gradle/

# 코드 생성 산출물 (코드젠으로 재생성 가능)
# [CUSTOMIZE] 프로젝트별 생성 코드 경로 추가
# **/generated/

# IDE 캐시
.idea/
*.iml
.vscode/

# 워크트리 (별도 세션에서 작업)
*_worktree/

# 기타
.DS_Store
coverage/
__pycache__/
*.pyc
```

### 2. 서브에이전트 모델 설정

탐색/분석용 서브에이전트에 적절한 모델을 지정.

```json
// .claude/settings.json
{
  "env": {
    "CLAUDE_CODE_SUBAGENT_MODEL": "sonnet"
  }
}
```

**모델 선택 기준:**

| 프로젝트 복잡도 | 서브에이전트 모델 | 이유 |
|---------------|----------------|------|
| 단순 (컨벤션 적음) | haiku | 탐색/요약이면 충분 |
| 보통 | sonnet | 패턴 이해 + 정확한 요약 |
| 복잡 (ADR/컨벤션 다수) | sonnet (최소) | haiku는 복잡한 규칙을 선택적으로 무시 |

### 3. MCP 서버 정리

각 MCP 서버는 도구 정의를 컨텍스트에 주입한다. 일부는 15,000+ 토큰을 차지.

- `/mcp`로 현재 서버 목록 확인
- 사용하지 않는 서버는 비활성화
- CLI 도구(`gh`, `aws`, `gcloud` 등)가 가능하면 MCP 서버 대신 사용 — CLI는 도구 정의가 컨텍스트에 추가되지 않음

## Tier 2: 습관으로 절약

### 4. `/compact` 선제 실행

컨텍스트 경고가 뜨기 전, 자연스러운 구분점에서 실행.

| 시점 | 이유 |
|------|------|
| Phase 전환 (plan → impl) | 이전 phase 컨텍스트 불필요 |
| git commit 직후 | 이전 작업 맥락 정리 |
| 대규모 코드 분석 완료 후 | 분석 결과만 남기고 원문 폐기 |
| 스킬 실행 완료 후 | 스킬 내부 컨텍스트 정리 |

보존할 내용을 명시:
```
/compact Phase 1 완료, Phase 2 진행 중. ADR-003 패턴 따르는 중
```

### 5. Escape + `/rewind` 즉시 손절

잘못된 방향을 **2K 토큰 이내에** 잡는다. 20K 토큰까지 가면 토큰 낭비 + 오염된 컨텍스트가 이후 턴에 영향.

| 신호 | 행동 |
|------|------|
| 엉뚱한 파일 읽기 시작 | Escape → `/rewind` |
| 컨벤션 무시하고 자기 방식으로 구현 | Escape → `/rewind` → 구체적 규칙 지시 |
| 스킬 없이 자유 생성 시작 | Escape → `/rewind` → 스킬 재호출 |

### 6. 프롬프트 정밀도

```
# 나쁨 — 탐색 범위 무제한
"이 코드 좀 개선해줘"

# 좋음 — 파일/함수 지정
"src/auth/login.ts의 handleLogin 함수에서 에러 핸들링을 추가해줘"
```

스킬이 이미 구체적 파일을 지정하므로, **스킬 외 직접 대화에서도 동일하게**.

### 7. 세션 관리

```
세션 1개 = 작업 1개
```

- 작업 전환 시 `/clear`로 새로 시작 (다른 작업의 컨텍스트 오염 방지)
- `--resume`보다 새 세션 선호 (캐시 미스 리스크 회피)
- worktree를 사용하면 세션 = worktree = 작업 1개로 자연스럽게 분리

## Tier 3: 아키텍처 수준 절약

### 8. 서브에이전트 활용 — "3파일 룰"

파일 3개 이상을 읽어야 하는 탐색/분석 작업은 서브에이전트에 위임.

**이유:** 서브에이전트는 부모 컨텍스트를 상속하므로 프롬프트 캐시 덕에 비용이 거의 동일. 내부에서 50K 토큰을 사용하더라도 요약 반환 후 폐기 → 메인 컨텍스트는 깨끗하게 유지.

```markdown
# CLAUDE.md에 추가 가능
## Context Management
Default to spawning agents for:
- Codebase exploration (reading 3+ files)
- Research tasks (web searches, doc lookups)
- Code review or analysis (verbose output)
```

### 9. Agent Teams는 신중하게

Agent Teams는 각 팀원이 **자체 컨텍스트 윈도우를 유지**하며, 단일 세션 대비 **약 7배 토큰을 소비**한다.

| 상황 | 추천 | 이유 |
|------|------|------|
| 독립적 병렬 작업 | 서브에이전트 | 조율 오버헤드 불필요 |
| 순차 작업 | 서브에이전트 | 컨텍스트 폐기, 저비용 |
| 팀원 간 소통 필요 | Agent Teams | 메일박스/태스크리스트 활용 |
| 대규모 모노레포 리팩터 | Agent Teams | 단일 컨텍스트 초과 |

Agent Teams 사용 시:
- 팀원 모델을 **Sonnet**으로 설정 (Opus 대비 약 1/5 비용)
- 팀 규모를 **최소한으로** 유지
- **작업 완료 즉시 팀 정리** — 유휴 팀원도 토큰 소비

### 10. 모델 전환 금지 (세션 중)

프롬프트 캐시는 모델별로 별도. 100K 토큰까지 쌓인 Opus 대화에서 Haiku로 바꾸면 캐시를 처음부터 새로 빌드 → 오히려 더 비싸다.

- 메인 대화에서 `/model` 전환 → 캐시 미스 → 비용 폭발
- Agent 도구에서 `model: "sonnet"` 지정 → 별도 컨텍스트이므로 안전

## CLAUDE.md 크기에 대한 판단

범용 가이드에서는 "500 토큰 이하"를 권장하지만, **하네스 프로젝트에서는 상황이 다르다.**

| 상황 | CLAUDE.md 크기 | 이유 |
|------|---------------|------|
| 신규/단순 프로젝트 | ~50줄 (500 토큰) | 컨벤션이 적으므로 최소화 |
| 컨벤션이 있는 프로젝트 | ~200줄 (task-conventions.md 분리) | standards + ADR로 계층화 |
| ADR/하네스 복잡한 프로젝트 | 현행 유지 | CLAUDE.md가 하네스의 핵심 구속 조건. 축소하면 Plan 품질 저하 → 검수 증가 → 총 토큰 증가 |

**판단 기준:** CLAUDE.md를 줄여서 절약하는 토큰(매 턴 ΔN)보다, 줄여서 늘어나는 검수 횟수(ΔR × 10K~30K)가 크면 줄이지 않는다.

마찬가지로 `MAX_THINKING_TOKENS` 제한도:
- 단순 작업: `/effort` 명령으로 일시적으로 낮추기 가능
- 복잡한 Plan 모드: 제한하면 Plan 품질 저하 → 역효과

## `/revise-claude-md`로 컨벤션 관리

Anthropic 공식 플러그인 `claude-md-management`를 사용하면 세션에서 배운 내용을 CLAUDE.md에 자동 반영할 수 있다.

```bash
# 설치
claude plugin install claude-md-management@claude-plugins-official
```

**제공 기능:**

| 커맨드/스킬 | 용도 | 시점 |
|-----------|------|------|
| `/revise-claude-md` | 세션에서 배운 내용을 CLAUDE.md에 반영 | 세션 종료 전 |
| `claude-md-improver` | CLAUDE.md 품질 감사 (100점 만점) | 정기 유지보수 |

**토큰 절약 효과:**
- 세션에서 발견한 패턴/규칙이 CLAUDE.md에 누적 → 다음 세션에서 재발견 불필요 → 탐색 토큰 절약
- 품질 감사로 CLAUDE.md를 간결하게 유지 → 매 턴 주입되는 토큰 최적화

## 체크리스트

### 즉시 적용
- [ ] `.claudeignore` 생성 (build, node_modules, IDE 캐시, 생성 코드)
- [ ] 사용하지 않는 MCP 서버 비활성화
- [ ] 서브에이전트 모델 설정 (`CLAUDE_CODE_SUBAGENT_MODEL`)

### 습관
- [ ] Phase 전환 / commit 후 `/compact` 선제 실행
- [ ] 잘못된 방향 감지 시 2K 토큰 이내에 Escape + `/rewind`
- [ ] 세션 1개 = 작업 1개
- [ ] 직접 대화에서도 파일/함수 경로 명시
- [ ] 세션 종료 전 `/revise-claude-md`로 학습 내용 반영

### 아키텍처
- [ ] 3파일+ 탐색은 서브에이전트에 위임
- [ ] Agent Teams는 팀원 간 소통이 진짜 필요한 경우에만
- [ ] 세션 중 `/model` 전환 금지

### 하지 말 것 (하네스 프로젝트)
- [ ] CLAUDE.md를 무작정 축소하지 않기 — 검수 횟수 ROI 계산
- [ ] `MAX_THINKING_TOKENS`를 복잡한 작업에서 제한하지 않기
- [ ] 복잡한 프로젝트에서 서브에이전트를 Haiku로 설정하지 않기
