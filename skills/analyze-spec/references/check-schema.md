# check-schema: DB 스키마 대조

기획 분석 MD의 데이터 항목을 실제 DB 스키마와 대조하여 매칭/누락/신규 필요 항목을 리포트한다.

## 진입 방식

1. `/analyze-spec check-schema` → AskUserQuestion으로 spec MD 경로 입력
2. `/analyze-spec check-schema docs/specs/xxx.md` → 지정된 경로 사용

---

## Step CS-1: 입력 준비

- spec MD 파일을 Read로 읽는다
- spec에서 관련 도메인을 판별한다

---

## Step CS-2: DB 스키마 로드

프로젝트의 DB 스키마 파일(ERD, DDL, 마이그레이션)을 탐색하여 읽는다.

탐색 경로 (프로젝트 구조에 따라 조정):
- `database/` — ERD(DBML), DDL 파일
- `migrations/` — 마이그레이션 파일
- `prisma/schema.prisma` — Prisma 스키마 (해당 시)

---

## Step CS-3: 대조 분석

spec MD에서 추출한 데이터 항목을 DB 스키마와 대조한다:

**분석 항목:**

1. **테이블 매칭** — spec에서 언급된 엔티티/자원이 어떤 테이블에 대응되는지
2. **컬럼 매칭** — spec의 필드/항목이 테이블의 어떤 컬럼에 매핑되는지
3. **누락 컬럼** — spec에는 있지만 DB에 없는 필드 (DDL 변경 필요)
4. **누락 테이블** — spec에 필요하지만 DB에 없는 테이블 (신규 DDL 필요)
5. **코드성 항목 매칭** — enum이나 코드 테이블 대조
6. **타입 불일치** — spec 명세와 DB 컬럼 타입이 안 맞는 경우

---

## Step CS-4: 리포트 생성

```markdown
## DB 스키마 대조 결과

### 요약
- **매칭 완료**: {N}건
- **누락 (DDL 변경 필요)**: {N}건
- **신규 테이블 필요**: {N}건
- **타입 불일치**: {N}건

### 테이블 매핑
| spec 항목 | DB 테이블 | 상태 |
|-----------|----------|------|
| {엔티티명} | `{table}` | 매칭 / 컬럼 차이 / 없음 |

### 컬럼 상세
#### {테이블명}
| spec 필드 | DB 컬럼 | 타입 | 상태 | 비고 |
|-----------|---------|------|------|------|

### 신규 필요 항목
#### DDL 변경
- [ ] `{테이블}.{컬럼}` 추가 — {사유}
- [ ] `{테이블}` 신규 생성 — {사유}
```

---

리포트를 spec MD와 같은 디렉토리에 저장:
- 파일명: `{spec-name}-schema-check.md`
